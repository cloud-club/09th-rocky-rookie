# RHEL 10 리눅스 스터디
## 컨테이너와 가상머신의 차이 이해
### Containers vs Virtual Machines on RHEL 10

---

## 0. 학습 목표 및 사전 지식

이 학습 자료는 RHEL 10 환경에서 시스템 가상화 기술을 다루기 전에 갖추어야 할 개념적 기반을 다집니다. 학습을 마치면 다음을 할 수 있어야 합니다.

- 컨테이너와 가상머신을 각각 정의하고, 서로의 핵심 구성 요소를 그림 없이도 설명할 수 있다.
- 두 기술이 OS 커널, 하드웨어, 격리, 부팅 흐름 측면에서 어떻게 다른지 비교할 수 있다.
- RHEL 10에서 사용하는 대표 도구(Podman, KVM/QEMU, libvirt)가 각각 어느 영역에 속하는지 구분할 수 있다.
- 주어진 워크로드 시나리오에서 컨테이너와 VM 중 무엇이 더 적합한지 근거를 들어 선택할 수 있다.

선행 지식으로는 리눅스 기본 명령어 사용 경험, 프로세스/파일시스템 개념, systemd 단위 파일에 대한 기초적 이해가 있으면 충분합니다.

## 1. 가상화 기술의 큰 그림

'가상화(virtualization)'는 물리 자원(CPU, 메모리, 디스크, 네트워크)을 추상화하여 마치 독립된 자원이 여러 개 있는 것처럼 보이게 만드는 기술의 총칭입니다. 이 큰 우산 아래 매우 다른 두 가지 접근 방식이 존재하며, 그것이 바로 이 자료의 주제인 가상머신(VM)과 컨테이너입니다.

### 1.1 가상화의 두 갈래

- **하드웨어 가상화(Hardware Virtualization)**: 물리 머신의 하드웨어 자체를 가상으로 흉내 내어 그 위에 완전한 운영체제를 통째로 올리는 방식. 대표 사례가 가상머신(VM)이다. RHEL 10에서는 KVM(Kernel-based Virtual Machine) 하이퍼바이저가 이 영역을 담당한다.
- **OS 수준 가상화(OS-level Virtualization)**: 운영체제 커널은 호스트의 것을 그대로 공유하면서, 프로세스 그룹 단위로 격리된 실행 환경을 만드는 방식. 대표 사례가 컨테이너이며, RHEL 10에서는 Podman이 표준 도구이다.

두 기술 모두 '격리된 환경에서 애플리케이션을 실행한다'는 결과는 비슷해 보이지만, 그 결과를 만들어내는 메커니즘과 비용 구조가 근본적으로 다릅니다. 이 차이를 정확히 이해하는 것이 본 학습의 핵심입니다.

### 1.2 RHEL 10이 제공하는 가상화 스택

| 계층 | 기술/도구 | 역할 |
|------|-----------|------|
| 하이퍼바이저(VM) | KVM + QEMU | 리눅스 커널 모듈로 동작하는 Type-1 형태의 하이퍼바이저. 게스트 VM에 가상 CPU/메모리/장치를 제공 |
| VM 관리 계층 | libvirt, virsh, Cockpit | VM의 생성·시작·정지·스냅샷 등을 표준 API/CLI/웹 UI로 제공 |
| 컨테이너 엔진 | Podman | 데몬리스(daemonless)·루트리스(rootless) 컨테이너 엔진. Docker CLI와 호환 |
| 컨테이너 빌드/배포 | Buildah, Skopeo | 이미지 빌드(Buildah), 레지스트리 간 이미지 복사·검사(Skopeo) |
| 컨테이너 격리 메커니즘 | namespaces, cgroups v2, SELinux, seccomp | 리눅스 커널이 제공하는 격리/자원 제한/보안 기본기 |

## 2. 가상머신(VM)의 기본 개념

### 2.1 정의

가상머신은 물리 컴퓨터 한 대를 소프트웨어로 흉내 낸 '가짜 컴퓨터'입니다. 이 가짜 컴퓨터에는 가상 CPU, 가상 메모리, 가상 디스크, 가상 NIC가 할당되며, 그 위에 BIOS/UEFI 펌웨어가 동작하고, 그 위에 게스트 운영체제(Guest OS) 커널이 설치되어 부팅됩니다. 결과적으로 하나의 물리 서버 위에 여러 대의 독립적인 컴퓨터가 동시에 살아 있는 듯한 환경이 만들어집니다.

### 2.2 핵심 구성 요소

- **하이퍼바이저(Hypervisor)**: 물리 하드웨어와 게스트 OS 사이에서 자원을 분배·중재하는 소프트웨어 계층. 'VMM(Virtual Machine Monitor)'이라고도 부른다. RHEL의 KVM은 리눅스 커널이 곧 하이퍼바이저가 되는 구조다.
- **게스트 OS(Guest OS)**: VM 내부에 설치되어 실행되는 완전한 운영체제. 호스트가 RHEL 10이라도 게스트로 Windows Server, Ubuntu, 다른 버전의 RHEL 등을 자유롭게 올릴 수 있다.
- **가상 디스크 이미지**: VM의 디스크 내용을 담는 파일. 일반적으로 qcow2(QEMU Copy-On-Write v2) 포맷을 사용하며, 스냅샷·씬 프로비저닝을 지원한다.
- **가상 장치(Virtual Devices)**: 가상 NIC, 가상 블록 장치, 가상 콘솔 등. virtio 드라이버를 사용하면 게스트 OS와 하이퍼바이저 간 통신이 매우 효율적이 된다.

### 2.3 하이퍼바이저의 두 가지 유형

| 구분 | Type-1 (Bare-metal) | Type-2 (Hosted) |
|------|---------------------|-----------------|
| 설치 위치 | 물리 하드웨어 위에 직접 | 기존 호스트 OS 위에 애플리케이션처럼 |
| 대표 제품 | KVM(리눅스 커널), VMware ESXi, Microsoft Hyper-V | VirtualBox, VMware Workstation |
| 성능 | 오버헤드가 작아 운영 환경에 적합 | 상대적으로 오버헤드가 큼, 개발/학습용 |
| RHEL의 위치 | KVM은 리눅스 커널의 일부로서 Type-1처럼 동작 | — |

> **참고: KVM은 정확히 어디에 속하는가?**
>
> KVM은 '리눅스 커널이 그 자체로 하이퍼바이저가 된다'는 독특한 구조라 Type-1과 Type-2의 경계에 있다는 평가를 받습니다. 다만 운영 관점에서는 베어메탈 RHEL에 KVM을 설치한 형태가 거의 Type-1과 동등한 성능을 내므로, 실무에서는 Type-1으로 분류하는 것이 일반적입니다.

### 2.4 RHEL 10에서 VM을 다루는 흐름

RHEL 10에서 VM을 운용할 때 가장 자주 사용되는 도구 체인은 다음과 같습니다.

```bash
# 가상화 패키지 그룹 설치 (RHEL 9+/10의 올바른 문법)
# dnf install @virt
# dnf install virt-install virt-viewer cockpit-machines

# libvirt 모듈형 데몬 활성화 (RHEL 10 기준)
# RHEL 9부터 모놀리식 libvirtd 대신 virtqemud 사용을 권장
# systemctl enable --now virtqemud

# 새 VM 생성 (RHEL 10 게스트 ISO로 설치)
# virt-install \
    --name rhel10-guest \
    --memory 4096 --vcpus 2 \
    --disk size=20,format=qcow2 \
    --cdrom /var/lib/libvirt/images/rhel-10.iso \
    --os-variant rhel10.0

# VM 상태 확인 / 시작 / 정지
# virsh list --all
# virsh start  rhel10-guest
# virsh shutdown rhel10-guest
```

> **RHEL 10의 libvirt 데몬 아키텍처**
>
> RHEL 9부터 libvirt는 단일 모놀리식 `libvirtd` 대신 역할별 모듈형 데몬(virtqemud, virtnetworkd, virtstoraged 등)으로 전환했습니다. RHEL 10에서는 KVM/QEMU 관련 기능을 담당하는 `virtqemud`가 기본 데몬입니다. `libvirtd`는 레거시 호환성 목적으로만 남아 있으므로 RHEL 10 신규 환경에서는 `virtqemud`를 사용하십시오.

위 명령들이 의미하는 것은 단순합니다. '물리 머신을 한 대 더 만든다'는 행위에 가까우며, 그래서 VM 한 대가 부팅되는 데 수십 초~수 분이 걸리고, 디스크 이미지도 수 GB에서 수십 GB에 달하게 됩니다.

## 3. 컨테이너의 기본 개념

### 3.1 정의

컨테이너는 '격리되어 실행되는 한 무리의 프로세스'입니다. 별도의 커널을 부팅하지 않고, 호스트(RHEL 10)의 리눅스 커널을 그대로 공유하면서, 자기만의 독립된 파일시스템·네트워크·프로세스 공간을 갖는 환경을 만들어 그 안에서 애플리케이션을 실행합니다. 사용자에게는 가벼운 미니 OS처럼 보이지만, 본질적으로는 '잘 격리된 프로세스'일 뿐입니다.

### 3.2 컨테이너를 만들어내는 리눅스 커널 기능

컨테이너는 새로운 발명품이 아니라, 리눅스 커널이 오랫동안 발전시켜 온 격리·자원 제어 기능들을 하나로 묶은 결과물입니다. 핵심은 다음 네 가지입니다.

- **Namespaces(네임스페이스)**: 프로세스에게 '나만의 시스템 뷰'를 제공한다. PID, 네트워크, 마운트, UTS(호스트네임), IPC, 사용자 등 종류별로 분리된다. 컨테이너 안의 PID 1은 호스트에서는 단지 평범한 자식 프로세스 하나에 불과하다.
- **Cgroups v2(컨트롤 그룹)**: 프로세스 그룹의 CPU·메모리·I/O 사용량을 제한·계측한다. RHEL 10은 cgroups v2를 기본으로 사용한다.
- **SELinux**: 컨테이너 프로세스에 별도의 보안 레이블을 붙여, 컨테이너가 호스트나 다른 컨테이너의 자원을 함부로 건드리지 못하게 강제 접근 통제(MAC)로 막는다. RHEL의 핵심 보안 차별점이다.
- **seccomp / capabilities**: 컨테이너 프로세스가 호출할 수 있는 시스템 콜과 가질 수 있는 리눅스 capability(특권)를 최소화하여 공격 표면을 줄인다.

> **핵심 포인트**
>
> "컨테이너 = 네임스페이스 + cgroups + 보안 정책(SELinux/seccomp/capabilities) + 컨테이너 이미지" 라는 공식을 외우면, 이후의 모든 동작이 자연스럽게 설명됩니다.

### 3.3 컨테이너 이미지와 OCI 표준

컨테이너 이미지는 '애플리케이션을 실행하는 데 필요한 모든 파일을 모아 둔 읽기 전용 템플릿'입니다. 보통 베이스 OS의 사용자 공간(userland)인 라이브러리·바이너리·설정 파일과 그 위의 애플리케이션이 한 덩어리로 묶여 있으며, 다음과 같은 특징을 가집니다.

- **계층(layer) 구조**: 변경된 파일만 새 레이어로 쌓아 올려 이미지 크기와 빌드 시간을 줄인다.
- **불변성(immutability)**: 한 번 만들어진 이미지는 그대로 보존되며, 어느 환경에서 실행해도 동일한 결과를 보장한다.
- **OCI(Open Container Initiative) 표준**: 이미지 포맷과 런타임이 표준화되어 Podman, Docker, Kubernetes 등 다양한 도구가 같은 이미지를 쓸 수 있다.

Red Hat은 RHEL 사용자 공간을 그대로 담은 UBI(Universal Base Image)를 별도 구독 없이 무료로 제공하므로, RHEL 10 환경에서 컨테이너 이미지를 만들 때 사실상의 표준 베이스로 사용됩니다. RHEL 10에서는 **UBI 10** 이미지(`ubi10`)가 이에 해당합니다.

### 3.4 RHEL 10에서 컨테이너를 다루는 흐름

RHEL 10에서는 Docker 대신 Podman이 기본 도구로 채택되어 있습니다. 명령어 인터페이스는 docker와 거의 동일해서 학습 부담이 거의 없습니다.

```bash
# Podman 설치 (대부분 기본 설치되어 있음)
$ sudo dnf install -y container-tools

# RHEL UBI 10 기반 이미지 가져오기 (RHEL 10 환경에는 UBI 10 사용)
$ podman pull registry.access.redhat.com/ubi10/ubi:latest

# 컨테이너 실행 (루트리스, 즉 일반 사용자 권한으로)
$ podman run --rm -it ubi10/ubi /bin/bash

# 백그라운드 실행, 포트 매핑 예시 (nginx)
$ podman run -d --name web -p 8080:80 \
    registry.access.redhat.com/ubi10/nginx

# 상태 확인 / 정지 / 삭제
$ podman ps
$ podman stop web
$ podman rm web
```

**Podman의 차별점 요약**

1. **Daemonless**: 별도의 백그라운드 데몬 없이 동작하므로 데몬 장애가 모든 컨테이너에 영향을 주지 않는다.
2. **Rootless**: 일반 사용자 계정으로 컨테이너를 실행할 수 있어 보안 사고 시 피해 범위가 줄어든다.
3. **systemd 친화적 (Quadlet)**: RHEL 10에 탑재된 Podman 5.x에서는 **Quadlet**으로 컨테이너를 systemd 단위 파일처럼 선언적으로 관리하는 것이 표준이다. 구버전의 `podman generate systemd`는 Podman 4.4부터 deprecated되어 RHEL 10에서는 사용하지 않는다.
4. **Docker 호환**: 명령어와 Dockerfile이 그대로 호환된다 (`alias docker=podman`으로도 사용 가능).

## 4. 컨테이너와 가상머신의 핵심 차이

이 장이 본 자료의 중심입니다. 두 기술의 차이를 다섯 가지 관점에서 차례로 살펴봅니다: (1) 격리 수준과 아키텍처, (2) 자원 사용·성능, (3) 보안 모델, (4) 이식성과 배포 단위, (5) 운영·라이프사이클.

### 4.1 한눈에 보는 비교표

| 관점 | 가상머신 (VM) | 컨테이너 (Container) |
|------|--------------|---------------------|
| 격리 단위 | 가상 하드웨어 + 게스트 OS 전체 | 프로세스 그룹 (네임스페이스) |
| 커널 | VM마다 독립된 커널을 부팅 | 호스트 커널을 모든 컨테이너가 공유 |
| 부팅 시간 | 수십 초 ~ 수 분 | 수십 밀리초 ~ 수 초 |
| 이미지 크기 | 보통 수 GB ~ 수십 GB | 수 MB ~ 수백 MB |
| 메모리 오버헤드 | 게스트 OS 분량(수백 MB~)이 항상 추가됨 | 프로세스 메모리 외에는 거의 없음 |
| 밀도(서버당 인스턴스 수) | 수십 대 수준 | 수백~수천 개 수준 |
| 격리 강도 | 강함 (하드웨어 경계) | 상대적으로 약함 (커널 경계) |
| OS 다양성 | Linux/Windows 등 자유롭게 혼합 | 호스트와 같은 커널 계열만 가능 (예: Linux 호스트→Linux 컨테이너) |
| 대표 도구(RHEL 10) | KVM/QEMU, libvirt, virsh | Podman, Buildah, Skopeo |
| 대표 사용 사례 | 레거시 OS, 강한 격리가 필요한 멀티테넌시, 풀스택 인프라 격리 | 마이크로서비스, CI/CD, 일관된 배포 단위, 빠른 스케일링 |

### 4.2 격리 수준과 아키텍처

VM과 컨테이너의 가장 근본적인 차이는 '무엇을 가상화하느냐'입니다. VM은 하드웨어를 가상화하고, 컨테이너는 OS의 사용자 공간을 가상화합니다.

**VM의 계층 구조 (위로 갈수록 추상화)**

```
┌──────────────────────────────────────────┐
│   App A     │   App B     │   App C      │
│   Bins/Libs │   Bins/Libs │   Bins/Libs  │
│   Guest OS  │   Guest OS  │   Guest OS   │  ← 각자 커널/OS
├──────────────────────────────────────────┤
│           Hypervisor (KVM/QEMU)          │
├──────────────────────────────────────────┤
│          Host OS (RHEL 10)               │
├──────────────────────────────────────────┤
│             Physical Hardware            │
└──────────────────────────────────────────┘
```

**컨테이너의 계층 구조**

```
┌──────────────────────────────────────────┐
│   App A     │   App B     │   App C      │
│   Bins/Libs │   Bins/Libs │   Bins/Libs  │  ← 사용자 공간만 격리
├──────────────────────────────────────────┤
│   Container Engine (Podman / crun)       │
├──────────────────────────────────────────┤
│   Host OS (RHEL 10) — 단일 커널 공유     │  ← namespaces/cgroups
├──────────────────────────────────────────┤
│             Physical Hardware            │
└──────────────────────────────────────────┘
```

VM은 게스트 OS마다 자기 커널을 부팅하므로 호스트와 게스트 사이의 경계가 '하드웨어 가상화 경계'입니다. 반면 컨테이너는 모두가 같은 호스트 커널을 공유하며, 경계는 '리눅스 커널이 만들어낸 논리적 경계(namespace/cgroup)'입니다. 이 차이가 이후 모든 다른 차이의 뿌리가 됩니다.

### 4.3 자원 사용과 성능

VM은 게스트 OS 자체가 자원을 소비합니다. 게스트 커널, systemd, 백그라운드 데몬, 가상 디스크 캐시 등이 모두 메모리와 CPU를 잡아먹기 때문에, 애플리케이션이 사용하지 않는 '기본 비용(baseline cost)'이 항상 깔립니다. 부팅 시 BIOS/UEFI → 부트로더 → 커널 → init → 사용자 서비스 순서를 모두 거쳐야 하므로 시작 시간도 길어집니다.

컨테이너는 호스트 커널이 이미 부팅되어 있으므로, 새 컨테이너를 시작한다는 것은 사실상 '새 프로세스를 fork/exec하면서 격리 옵션을 부여하는 일'에 가깝습니다. 이미지에 들어 있는 사용자 공간 파일을 마운트하고, 격리된 PID/네트워크 네임스페이스 안에서 첫 프로세스를 실행하면 끝입니다. 그래서 시작이 거의 즉시이고, 메모리 오버헤드도 미미합니다.

| 항목 | VM | 컨테이너 |
|------|-----|---------|
| 일반적 시작 시간 | 30초 ~ 수 분 | 0.1초 ~ 수 초 |
| 디스크 사용량(이미지 1개) | 수 GB ~ 수십 GB | 수십 MB ~ 수백 MB |
| 기본 메모리 점유(앱 외) | 수백 MB ~ 수 GB | 거의 0 (애플리케이션 메모리만) |
| CPU 오버헤드 | 낮음(virtio/하드웨어 가상화 지원 시)이지만 0은 아님 | 사실상 호스트 네이티브 수준 |
| I/O 성능 | virtio·SR-IOV 등 사용 시 높지만 계층 통과 비용 존재 | 호스트 파일시스템·네트워크 스택 직접 사용 |

### 4.4 보안 모델: '무엇을 신뢰하는가'

보안에서는 '공격자가 격리를 뚫고 나갔을 때 어디까지 갈 수 있는가'가 핵심 질문입니다.

- **VM의 보안 경계**: 공격자가 게스트 OS 안에서 root 권한을 얻더라도, 호스트로 빠져나가려면 하이퍼바이저(또는 가상 장치 에뮬레이션)의 취약점을 추가로 뚫어야 한다. 이 경계는 매우 두꺼워 'VM 탈출(VM escape)' 취약점은 매우 드물고 보통 큰 사건으로 다뤄진다.
- **컨테이너의 보안 경계**: 모든 컨테이너가 호스트 커널을 공유하므로, 커널 자체의 취약점이 컨테이너 탈출 통로가 될 수 있다. 그래서 '컨테이너 격리는 VM만큼 강하지 않다'는 말이 자주 나온다.

그러나 RHEL 10에서는 컨테이너 보안을 다층(defense-in-depth)으로 보강합니다.

- **SELinux**: `container_t` 같은 전용 도메인을 컨테이너 프로세스에 적용해, 호스트 파일시스템과의 상호작용을 강력히 제한한다.
- **Rootless 컨테이너**: Podman 기본 동작. 컨테이너 안의 root가 호스트에서는 비특권 사용자에 매핑되어, 탈출하더라도 즉시 호스트 root가 되지 못한다.
- **seccomp 프로파일과 capabilities 최소화**: 위험한 시스템 콜과 권한을 기본적으로 차단한다.
- **이미지 신뢰성**: Red Hat 인증 이미지(레지스트리 서명·취약점 스캔 포함)를 사용하면 공급망 보안 위험을 낮출 수 있다.

> **실무 가이드라인**
>
> 외부에서 들어온 신뢰할 수 없는 코드를 격리해야 한다면 VM이 더 안전한 선택입니다. 반대로, 신뢰할 수 있는 자체 애플리케이션을 효율적으로 패키징·배포하는 것이 목적이라면 컨테이너의 격리 강도는 SELinux와 rootless 운영을 결합했을 때 충분히 합리적인 수준입니다.

### 4.5 이식성과 배포 단위

VM의 '배포 단위'는 디스크 이미지(qcow2 등)이며, 게스트 OS 전체가 들어 있어 크고 무겁습니다. 다른 하이퍼바이저로 옮길 때는 포맷 변환이 필요할 수 있고, 부팅 환경(펌웨어, 가상 칩셋)이 달라지면 부팅이 실패할 수도 있습니다.

컨테이너의 배포 단위는 OCI 이미지입니다. 같은 이미지를 개발자 노트북, CI 서버, 스테이징, 운영 환경에서 동일하게 실행할 수 있다는 점이 가장 큰 장점입니다. '내 PC에서는 되는데요(works on my machine)' 문제가 구조적으로 사라지며, 이것이 컨테이너가 DevOps와 마이크로서비스의 표준 배포 수단이 된 가장 큰 이유입니다.

### 4.6 운영과 라이프사이클

두 기술은 운영 모델 자체가 다릅니다.

- **VM**: '오래 사는 가축'에 가깝다. 한 번 만들어 패치를 누적하고, 백업·스냅샷으로 보존하며, 수개월~수년 동안 유지된다. 라이브 마이그레이션, HA 클러스터 등 전통 인프라 운영 패턴이 그대로 적용된다.
- **컨테이너**: '짧게 사는 일회용'에 가깝다. 새 버전이 나오면 기존 컨테이너를 폐기하고 새 이미지로 다시 띄우는 것이 표준이다. 상태(state)는 컨테이너 밖의 볼륨이나 외부 데이터베이스에 저장한다(이른바 '12-Factor App' 사고방식).

그래서 컨테이너 운영은 자연스럽게 오케스트레이션(예: Kubernetes/OpenShift)과 결합되며, RHEL 10은 OpenShift 노드 OS인 RHCOS와 동일한 컨테이너 도구 체인을 공유합니다.

## 5. 컨테이너 vs VM이 아니라, '함께 쓰기'

실무에서는 둘 중 하나만 선택하기보다는 두 기술을 계층적으로 결합하는 경우가 더 많습니다. 가장 흔한 패턴은 다음과 같습니다.

- **'VM 위에 컨테이너'**: 클라우드/온프레미스의 가상화 인프라가 제공한 RHEL 10 VM에 Podman을 설치하고 그 안에서 컨테이너 워크로드를 실행한다. VM은 강한 멀티테넌시 격리를, 컨테이너는 빠른 배포와 밀도를 담당한다.
- **'OpenShift/Kubernetes 클러스터'**: 노드는 VM이거나 베어메탈일 수 있으며, 그 위에서 모든 애플리케이션은 컨테이너(Pod)로 실행된다. RHEL 10과 RHCOS가 노드 OS 역할을 한다.
- **'레거시는 VM, 신규는 컨테이너'**: 윈도우 기반 또는 OS 의존성이 큰 레거시 애플리케이션은 VM으로 두고, 새 마이크로서비스는 컨테이너화하여 점진적으로 현대화한다.

## 6. 무엇을 언제 선택할 것인가 — 의사결정 가이드

아래 표는 일반적인 선택 기준입니다. 실제 결정에서는 보안 요구 수준, 운영 인력의 숙련도, 기존 자산까지 종합 고려해야 합니다.

| 요구사항/시나리오 | 권장 | 이유 |
|----------------|------|------|
| 호스트와 다른 OS(예: Windows) 실행 필요 | VM | 컨테이너는 호스트 커널을 공유하므로 다른 OS 커널은 실행 불가 |
| 커널 모듈 로드, 시스템 콜 깊은 수준 사용 | VM | 컨테이너는 호스트 커널을 공유해 자유로운 커널 변경 불가 |
| 멀티테넌시(서로 신뢰하지 않는 사용자) 격리 | VM (또는 VM+컨테이너) | 하드웨어 가상화 경계가 더 두텁다 |
| 수백 개 마이크로서비스를 빠르게 스케일 | 컨테이너 | 시작 시간·자원 효율·이미지 이식성 모두 유리 |
| CI/CD 파이프라인의 일관된 빌드/테스트 환경 | 컨테이너 | 동일 이미지가 개발·테스트·운영에서 그대로 동작 |
| GPU/특수 하드웨어를 한 워크로드가 점유 | VM 또는 베어메탈 | 장치 패스스루(VFIO 등)와 결합한 VM 격리가 일반적 |
| 레거시 모놀리식 애플리케이션 그대로 운영 | VM | OS 종속성·시스템 서비스 의존성을 그대로 보존 |
| Edge·IoT처럼 자원 제약 환경 | 컨테이너 | 메모리·디스크 풋프린트가 작다 |

## 7. 자주 듣는 오해와 정정

**오해 1. '컨테이너는 가벼운 VM이다.'**

아닙니다. 컨테이너는 '격리된 프로세스'이지, 작아진 가상머신이 아닙니다. 자기 커널을 가지지 않고 호스트 커널을 공유한다는 점이 단순한 크기 차이를 넘어서는 본질적 차이입니다.

**오해 2. 'RHEL에서 컨테이너를 쓰려면 Docker가 필요하다.'**

아닙니다. RHEL 8 이후로 Docker는 기본 제공되지 않으며, RHEL 10에서는 Podman, Buildah, Skopeo가 표준입니다. 명령어 호환성이 매우 높아 Docker 사용 경험을 그대로 살릴 수 있습니다.

**오해 3. '컨테이너는 무조건 빠르고 VM은 무조건 느리다.'**

워크로드의 특성에 따라 다릅니다. CPU·메모리 위주의 잘 짜인 VM은 호스트와 거의 동등한 성능을 내며, 거꾸로 컨테이너도 네트워크 오버레이나 스토리지 드라이버 선택에 따라 성능 손실이 생길 수 있습니다. '기본 비용'과 '부팅 시간' 측면에서 컨테이너가 유리한 것이 일반적이라는 정도로 이해하면 충분합니다.

**오해 4. '컨테이너가 보안적으로 항상 약하다.'**

커널 공유 모델 자체는 VM보다 격리 경계가 얇은 것이 사실이지만, RHEL의 SELinux + rootless Podman + seccomp + 검증된 UBI 이미지를 결합하면 매우 견고한 운영이 가능합니다. '약하다'가 아니라 '다른 모델이며, 다른 방식으로 강화해야 한다'가 정확한 표현입니다.

## 8. 정리

> **핵심 요약**
>
> 가상머신은 '하드웨어를 가상화하여 별도의 OS 커널을 부팅하는 기술'이며, 컨테이너는 '호스트 OS 커널을 공유한 채 격리된 프로세스 환경을 제공하는 기술'이다. 두 기술은 경쟁 관계가 아니라 서로 다른 문제를 푸는 도구이며, RHEL 10에서는 KVM/libvirt와 Podman이 각 영역의 표준 도구로 함께 제공된다.

스터디에서 이 자료를 읽은 뒤에는, 다음 단계로 다음 두 가지 실습을 진행하면 개념이 자연스럽게 몸에 붙습니다.

- **실습 A**: `virt-install`로 RHEL 10 게스트 VM을 한 대 만들고 부팅 시간·디스크 크기를 측정하기
- **실습 B**: 같은 애플리케이션을 `podman run`으로 컨테이너로 실행한 뒤, 시작 시간·메모리 사용량을 비교하기

두 결과를 직접 비교해 보면, 본 자료에서 설명한 '격리 수준의 차이'가 왜 '성능과 운영 모델의 차이'로 이어지는지가 가장 빠르게 이해됩니다.

| 용어 | 설명 |
|------|------|
| KVM | Kernel-based Virtual Machine. 리눅스 커널 모듈로 동작하는 하이퍼바이저 |
| QEMU | 장치 에뮬레이션을 담당하는 사용자 공간 컴포넌트. KVM과 함께 동작 |
| libvirt | 여러 하이퍼바이저를 추상화하는 관리 API와 도구 모음. virsh CLI 포함 |
| virtqemud | RHEL 9+의 모듈형 libvirt 데몬. KVM/QEMU 하이퍼바이저 관리 전담. RHEL 10에서 libvirtd를 대체 |
| Podman | RHEL의 표준 컨테이너 엔진. 데몬리스·루트리스 지원 |
| Buildah | 컨테이너 이미지를 빌드하는 도구 (Dockerfile 또는 스크립트 방식) |
| Skopeo | 이미지 검사·복사 도구. 레지스트리 간 이미지 이동에 사용 |
| UBI | Universal Base Image. Red Hat이 제공하는 무료 재배포 가능 RHEL 베이스 이미지. RHEL 10 환경에서는 UBI 10(`ubi10`) 사용 |
| Namespace | 프로세스에 격리된 시스템 뷰를 제공하는 리눅스 커널 기능 |
| Cgroups v2 | 프로세스 그룹의 자원 사용을 제한·계측하는 통합 컨트롤 그룹 인터페이스 |
| OCI | Open Container Initiative. 컨테이너 이미지·런타임 표준 |
| Quadlet | 컨테이너를 systemd 단위 파일로 선언적으로 관리하는 Podman 통합 기능. RHEL 10(Podman 5.x)에서 표준 방식 |
