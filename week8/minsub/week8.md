## 8주차 스터디 개요
> RHEL 환경에서 컨테이너의 기본 개념을 이해하고, 이미지와 컨테이너의 차이를 학습한 뒤 Podman 또는 Docker를 활용한 기본 컨테이너 실행 실습을 진행하고자 한다.

그동안 RHEL의 기본 환경과 명령어, 패키지 및 사용자/권한 관리, 파일 시스템과 스토리지 관리, 서비스와 프로세스 제어, 네트워크 및 보안의 기초를 학습했다. 이번 주차에서는 `컨테이너 기반 실행 환경의 개념과 기본 사용법을 익히는 것`을 목표로 한다.

서버 및 애플리케이션 운영 환경에서는 일반적으로 애플리케이션을 컨테이너 단위로 실행하고 배포하는 방식을 사용한다. 컨테이너는 기존 가상머신보다 가볍고, 빠르게 실행할 수 있으며, 애플리케이션 실행 환경을 일관되게 유지할 수 있는 장점이 있는데, 아래에서 더 자세히 살펴보자.

---
## 1. 컨테이너란?

컨테이너는 애플리케이션과 그 실행에 필요한 라이브러리, 설정 파일 등을 하나로 묶어 실행할 수 있게 하는 기술이다.

즉, 애플리케이션이 어떤 서버에서 실행되든 동일한 환경에서 동작할 수 있도록 만들어주는 실행 단위로 볼 수 있다.

운영체제 전체를 새로 띄우는 것이 아니라, 호스트 운영체제의 커널을 공유하면서 독립된 실행 환경을 제공한다. 이 때문에 가상머신보다 훨씬 가볍고 빠르게 실행할 수 있다.

---
## 2. 가상머신과 컨테이너의 차이

### 가상머신 (VM)

가상머신은 하이퍼바이저 위에서 운영체제 전체를 별도로 실행하는 방식이다. 각 VM은 독립적인 커널과 운영체제를 가지므로 격리 수준은 높지만, 리소스를 더 많이 사용한다.

### 컨테이너 (Container)

컨테이너는 호스트의 커널을 공유한 채, 프로세스와 파일 시스템을 분리해 실행한다.

즉, 운영체제 전체를 부팅하지 않고 애플리케이션 실행에 필요한 환경만 제공한다.

그래서 VM은 무겁지만, 완전한 OS 단위를 격리할 수 있고 컨테이너는 가볍고 빠르지만 커널을 공유한다는 특징이 있다.

---

## 3. Docker와 Podman

컨테이너를 다루는 대표적인 도구로 Docker와 Podman이 있다.

### Docker

Docker는 가장 널리 알려진 컨테이너 도구로, 이미지 빌드, 컨테이너 실행, 네트워크, 볼륨 관리 기능 등을 제공한다.

이 모든 관리는 `Docker Daemon`을 통해 이루어지는데, Docker 시스템에서 중추적인 역할을 하는 백그라운드 프로세스이다.

기본적으로 데몬은 클라이언트의 명령을 수신하고, 이를 처리해서 컨테이너의 생성을 포함한 다양한 작업을 수행한다.

클라이언트의 명령을 받고 이를 실행하는 서버 역할을 하기 때문에, 사용자 인터페이스와는 별개의 존재이다. 데몬은 시스템이 부팅될 때 자동으로 시작되며, 지속적으로 백그라운드에서 동작하면서 Docker 관련 명령을 처리한다.

세부적인 주요 담당 기능은 다음과 같다.

### 1. 컨테이너 관리

Dokcer 데몬의 가장 핵심적 기능은 컨테이너의 생성, 실행, 중지, 삭제를 관리하는 것이다. 클라이언트가 docker run, docker stop, docker rm 등의 명령을 실행하면, 실제로는 Docker 클라이언트가 데몬에게 해당 작업을 요청하고, 데몬이 이를 처리하게 된다.

예를 들어 docker run 명령이 실행되면 Docker 데몬은 다음과 같은 작업을 처리한다.

- 지정된 이미지를 기반으로 컨테이너를 생성
- 필요한 네트워크 및 저장소 설정을 초기화
- 컨테이너 내부에서 프로세스를 시작
- 컨테이너 실행 상태를 지속적으로 모니터링

### 2. 이미지 관리

Docker 데몬은 이미지의 생성, 삭제, 저장소와의 통신을 담당한다. 이미지는 컨테이너의 청사진으로, Docker 데몬은 레지스트리(예: Docker Hub)에서 이미지를 풀링하거나 푸시하는 작업을 수행한다.

이미지가 존재하지 않는 상태에서 컨테이너를 생성하려고 하면, 데몬은 자동으로 레지스트리에서 이미지를 가져와 로컬 시스템에 저장한 후, 그 이미지를 사용해 컨테이너를 생성한다.

### 3. 네트워킹 관리

Docker 데몬은 Docker 컨테이너 간의 네트워킹을 관리한다. 각각의 컨테이너는 기본적으로 분리된 네트워크 환경을 가지며, Docker 데몬은 이런 네트워크 설정을 처리한다. 데몬은 컨테이너에 IP 주소를 할당하고, 브리지 네트워크를 생성하며, 컨테이너 간에 안전하게 통신할 수 있도록 하기 위해 네트워크를 격리한다.

### 4. 볼륨 관리

데몬은 또한 볼륨 관리로 컨테이너에 지속적인 저장소를 제공한다. 컨테이너가 중지되거나, 삭제되더라도 데이터를 유지하기 위해 볼륨을 사용하며, 이러한 볼륨은 Docker 데몬에 의해 생성되고 관리된다.

### 서버 - 클라이언트 구조

Docker의 아키텍처는 클라이언트 - 서버 모델을 따른다. 이 모델에서는 Docker 클라이언트가 docker 명령을 입력하면, 클라이언트는 Docker 데몬과 통신하여 명령을 전달하고, 데몬은 이를 처리한 후 클라이언트에 응답을 보낸다.

데몬은 클라이언트와의 통신을 위해 Unix 소켓 또는 네트워크 인터페이스를 사용한다. 예를 들어, 로컬에서 실행되는 Docker 클라이언트는 Unix 소켓을 통해 데몬과 통신하고, 원격으로 Docker 데몬과 통신할 때는 TCP 소켓을 사용할 수 있다.

- Unix 소켓: 로컬에서 Docker 클라이언트와 데몬 간의 기본적 통신 방법
- TCP 소켓: 원격 시스템에서 Docker 클라이언트와 데몬 간의 통신 방법



### API 기반 통신

Docker 데몬은 REST API를 통해 명령을 수신하고 처리한다.

Docker 클라이언트가 보내는 명령은 API 요청으로 변환되며, Docker 데몬은 이러한 요청을 처리한다. 예를 들어, docker run 명령은 실제로 데몬에게 컨테이너 생성을 요청하는 API 호출로 변환된다.

Docker API는 다양한 HTTP 메서드를 통해 데몬과 상호작용할 수 있게 하며, 외부 애플리케이션도 이를 통해 Docker를 제어할 수 있다.

### 리소스 관리

Docker 데몬은 Linux 커널의 기능인 네임스페이스(Namespace)와 컨트롤 그룹(Cgroup)을 사용하여 각 컨테이너의 리소스를 관리하고 격리한다. 이를 통해 컨테이너는 서로 간섭 없이 독립적으로 실행될 수 있으며, 호스트 시스템의 자원을 효율적으로 사용할 수 있다.

- 네임스페이스: 프로세스, 파일 시스템, 네트워크 등 시스템의 다양한 자원을 격리하여 각 컨테이너가 독립적인 실행 환경을 가질 수 있도록 해줌
- 컨트롤 그룹(Cgroups): CPU, 메모리, 디스크 I/O, 네트워크 대역폭 등 각 컨테이너가 사용할 수 있는 시스템 자원을 제한한다.

### Docker 데몬과 컨테이너의 상태 관리

데몬은 컨테이너의 상태를 지속적으로 모니터링하고 관리한다.

데몬은 각 컨테이너의 시작, 정지, 재시작, 삭제 등의 작업을 처리하며, 하기의 상태를 가질 수 있다.

- created: 컨테이너가 생성되었지만, 아직 시작되지 않은 상태
- running: 컨테이너가 정상적으로 실행 중인 상태
- paused: 컨테이너의 모든 프로세스가 일시 중지된 상태
- stopped: 컨테이너가 중지된 상태

### 컨테이너 재시작 정책

데몬은 컨테이너의 재시작 정책도 관리하는데, 예를 들어 애플리케이션이 비정상적으로 종료되었을 때, 데몬은 설정된 재시작 정책에 따라 컨테이너를 자동으로 다시 실행할 수 있다.

`docker run --restart always my-cloudclub-app`

위 명령은 컨테이너가 예기치 않게 종료되더라도 데몬이 자동으로 컨테이너를 재시작하도록 설정하는 명령어 예시이다.

### Docker 데몬의 설치와 관리

데몬은 대부분 리눅스 배포판에서 쉽게 설치할 수 있고, 설정 파일을 통해 동작 방식을 세밀하게 조정할 수 있다. 보통 systemd와 같은 서비스 매니저를 통해 관리되며, systemctl 명령어로 시작, 중지, 재시작할 수 있다.

- `systemctl status docker` : 데몬 상태 확인
- `journalctl -u docker.service` : 데몬 로그 확인 (컨테이너 생성, 중지, 오류 발생 등의 정보 파악 가능)

### 보안

데몬은 시스템 자원에 대한 접근 권한을 가지고 있어, 보안 관리가 중요하다. 데몬을 보호하기 위해 하기의 보안 대책을 적용할 수 있다.

- `루트 권한 최소화`: 데몬은 루트 권한으로 실행되므로, 컨테이너가 데몬과 상호 작용할 때 보안 위협이 발생할 수 있는데, 이를 방지하기 위해 **rootless mode**를 사용해서 루트 권한 없이 실행할 수도 있다.
- `TLS를 통한 보안 통신`: 데몬과 클라이언트 간 통신은 기본적으로 보호되지 않아, TLS 인증서를 사용하여 보안을 강화할 수 있다. 특히, 원격에서 데몬에 접근할 때는 반드시 TLS 설정을 적용해야 한다.

결론적으로 Docker 데몬은 Docker 플랫폼의 핵심 구성 요소로, 컨테이너와 관련된 모든 작업을 처리하고 관리하는 중요한 역할을 한다. 데몬은 네트워킹, 저장소, 상태 관리 등을 담당하며, 클라이언트와의 통신에서 다양한 명령을 처리한다. Docker를 제대로 활용하려면 데몬의 동작 방식을 이해하고 이를 적절하게 관리하는 것이 매우 중요하다.

### Podman

Docker와는 달리 Podman은 데몬 없이 동작할 수 있는 컨테이나 관리 도구이며, RHEL 계열 환경에서 많이 사용된다.

기본적인 명령어 구조가 Docker와 매우 유사해서 Docker를 익힌 사용자도 비교적 쉽게 사용할 수 있다.

실습 환경이 RHEL이라면 Podman 중심으로 학습하는 것이 자연스럽고, Docker와 개념은 거의 유사하다.

### Podman에서 Pod란?

Podman에서 포드는 쿠버네티스 포드와 유사하게 함께 실행되고 동일한 리소스를 공유하는 컨테이너 그룹이다. Podman은 컨테이너, 포드, 컨테이너 이미지 및 볼륨을 관리하기 위한 API를 제공하는 libpod 라이브러리와 간단한 CLI를 통해 이러한 포드를 관리한다. Podman의 CLI는 컨테이너 런타임과 형식을 위한 업계 표준을 준수하도록 설계된 Open Container Initiative(OCI) 컨테이너를 생성하고 지원한다.

각 포드는 하나의 인프라 컨테이너와 다수의 일반 컨테이너로 구성되어 있고, 인프라 컨테이너는 포드를 계속 실행하며, 호스트에서 컨테이너를 격리하는 사용자 네임스페이스를 유지한다. 다른 일반 컨테이너는 자체 프로세스를 추적하고 종료된 컨테이너, 즉 리소스 일부가 여전히 사용되기 때문에 환경에서 제거할 수 없으며, 작동하지 않는 컨테이너를 찾는 모니터를 각각 갖추고 있다.

### Podman과 다른 컨테이너 엔진의 차이점

podman은 앞서 언급한 것처럼, 데몬이 없다는 점에서 타 엔진과 차별점을 가진다. 데몬이 없다는 것은 컨테이너를 실행할 때 루트 권한이 있는 프로세스에 의존하지 않는다는 뜻이다.

데몬은 컨테이너 환경을 편리하게 관리할 수 있지만, 보안 취약점이 발생할 수 있었다. Linux 시스템에서 루트 계정은 관리자 액세스 권한이 있는(동시에 관리자 검증 절차를 건너뛰는) 슈퍼 사용자 역할을 하면서 파일을 읽고, 프로그램을 설치하고, 애플리케이션을 수정한다. 따라서, 데몬은 컨테이너를 장악하고, 호스트 시스템에 잠입하려는 해커들에게 매우 좋은 표적이 될 수 있다.

그러나, Podman은 데몬을 제거하고 일반 사용자가 루트를 보유한 데몬과 상호 작용할 필요 없이 컨테이너를 실행하거나 루트리스 컨테이너의 사용을 허용한다. 루트리스로 전환을 하면 사용자는 관리자 권한이 있는 프로세스 없이도 컨테이너를 생성, 실행, 관리할 수 있으므로 컨테이너 환경에 대한 접근성을 높이는 동시에 보안 위험을 줄일 수 있다. 또한, Podman은 각각 Security-Enhanced Linux(SELinux) 레이블이 할당된 컨테이너를 실행하므로 관리자는 컨테이너 프로세스에 제공되는 리소스와 기능을 더욱 세밀하게 제어할 수 있다.

### 그렇다면 Podman은 어떻게 컨테이너를 관리할까?

사용자는 커맨드라인에서 Podman을 호출하여 리포지토리에서 컨테이너를 가져와 실행할 수 있다. 구성된 컨테이너 런타임을 호출하여 실행 중인 컨테이너를 생성하고, systemd를 사용하여 업데이트하고 백그라운드에서 컨테이너를 지속적으로 실행한다. systemd와 Podman을 통합하면 컨테이너에 대한 제어 장치를 생성하고, systemd가 자동으로 활성화된 상태로 해당 제어 장치를 실행할 수 있다.

사용자는 시스템에 있는 리포지토리 또는 systemd 유닛을 통해 컨테이너의 자동 시작과 관리를 제어할 수 있는데, 사용자가 자신의 리소스를 관리하고 루트리스 컨테이너를 실행하면, 특정 권한을 추가하려 할 때 해당 권한이 비허용되는 시스템 영역에 제한을 둘 수 있다. 즉 모든 사용자가 별도의 컨테이너와 이미지 세트를 보유하고, 동일한 호스트에서 서로 간섭하지 않고 Podman을 사용할 수 있다. 사용자가 작업을 마치면 공통 레지스트리로 변경 사항을 푸시하여 다른 사람과 이미지를 공유할 수 있다.

## 이미지와 컨테이너의 차이

컨테이너를 이해할 때 가장 먼저 구분해야 할 개념은 이미지와 컨테이너이다.

### 이미지 (Image)

이미지는 애플리케이션 실행에 필요한 파일과 설정을 담고 있는 `읽기 전용 템플릿`이다.

예를 들어, `nginx, alpine, ubuntu` 같은 이미지를 내려받아 사용할 수 있다.

### 컨테이너 (Container)

컨테이너는 이미지를 실제로 실행한 인스턴스이다.

즉, 이미지는 실행 전의 설계도이고, 컨테이너는 그 설계도를 기반으로 실제로 실행 중인 실체라고 볼 수 있다.

예를 들어 `nginx`  이미지를 다운로드하고 그 이미지로 컨테이너를 실행한 후에, 컨테이너가 웹 서버 프로세스 내부에서 동작하도록 하는 일련의 흐름으로 이해할 수 있다.

## 실습

## 1. 컨테이너 런타임 확인

RHEL 환경에서는 먼저 Podman이 설치되어 있는지 확인했다.

## 2. 이미지 검색 및 다운로드 실습

### 2-1. 이미지 검색

먼저 사용할 이미지를 검색했다.

- `podman search [검색할 이미지명]` : 실습에서는 apline 관련 이미지를 검색했다.
- 컨테이너 레지스트리에서 사용할 수 있는 이미지 목록을 확인할 수 있다.

## 2-2. 이미지 다운로드

- `podman pull alpine` : 컨테이너 이미지 다운로드

로컬 환경에 `alpine` 이미지를 저장한다.

## 2-3. 다운로드된 이미지 확인

- `podman images` : 현재 로컬에 저장된 이미지 목록 확인

여기서 이미지 이름, 태그, 이미지 ID, 크기 등을 확인한다.

## 3. 컨테이너 실행 실습

### 3-1. 간단한 컨테이너 실행

다운로드한 이미지를 사용해 컨테이너를 실행했다.

- `podman run` : 이미지를 기반으로 새 컨테이너 실행
- `echo "hello container"` : 컨테이너 안에서 실행할 명령

### 3-2. 대화형 컨테이너 실행

컨테이너 내부 셸에 직접 들어가기 위해 대화형 모드로 실행했다.

- `podman run -it alpine /bin/sh`
- `-i` : 표준 입력 유지
- `-t` : 터미널 연결
- `/bin/sh` : 컨테이너 내부에서 실행할 셸

### 3-3. 실행 이력 확인

- `podman ps -a` : 실행 중이거나 종료된 모든 컨테이너 확인

## 4. 컨테이너 시작, 중지, 삭제

### 4-1. 백그라운드 실행

컨테이너를 백그라운드에서 실행해보자.

- `-d`  : 백그라운드 실행
- `--name myalpine` : 컨테이너 이름 지정
- `sleep 1000` : 1000초 동안 실행 유지

실행 후 확인:

- `podman ps` : 현재 실행 중인 컨테이너 확인

### 4-2. 컨테이너 중지

- `podman stop` : 실행 중인 컨테이너 중지

중지 후 다시 확인

### 4-3. 컨테이너 삭제

- `podman rm` : 중지된 컨테이너 삭제

상기의 실습 과정들을 통해 컨테이너는 이미지와 별개로 생성/중지/삭제되는 실행 단위임을 확인했다.

## 5. 포트 매핑 실습

컨테이너 내부 서비스에 호스트에서 접근하려면 포트 매핑이 필요하다.

실습용으로 `nginx` 컨테이너를 실행했다.

- `p 8080:80` : 호스트의 8080 포트를 컨테이너의 80포트와 연결

## 6. 볼륨 연결

컨테이너는 기본적으로 일시적인 실행 환경이므로, 데이터를 지속적으로 보존하려면 볼륨 연결이 필요하다.

먼저 호스트에 디렉터리를 하나 생성한다.

- `mkdir -p ~/html` : 디렉터리 생성
- `podman run -d --name volume-nginx -p 8081:80 -v ~/html:/usr/share/nginx/html:Z  nginx` : -v 호스트경로:컨테이너경로 → 볼륨 연결
- `:Z` : SELinux 환경에서 적절한 보안 컨텍스트 부여

확인:

`curl http://localhost:8081`

상기 과정으로 호스트의 파일이 컨테이너 내부 웹 루트와 연결되어, 컨테이너를 재생성하더라도 데이터는 호스트에 남는다는 점을 확인했다.

## 7. 컨테이너 네트워크 기초 확인

실행 중인 컨테이너의 네트워크 정보를 확인하고자 했다.

- `podman inspect` : 컨테이너 상세 정보 확인
- `podman port` : 포트 매핑 정보 확인

간단히 네트워크 관련 정보가 어떻게 저장됐는지 정도로 진행했다.

## 8. 이미지와 컨테이너 정리

실습 후 사용한 이미지와 컨테이너를 정리했다.

### 컨테이너 정지 및 삭제

- `podman stop mynginx volume-nginx`
- `podman rm mynginx volume-nginx`

### 이미지 삭제

- `podman rmi` : 이미지 삭제

단, 해당 이미지를 사용 중인 컨테이너가 남아 있으면 이미 삭제 실패 가능

이번 마지막 주차 실습으로 하기의 내용을 익혔다.

- 컨테이너와 가상머신 차이
- 이미지와 컨테이너의 개념 차이
