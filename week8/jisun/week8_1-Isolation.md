# 컨테이너와 가상머신의 차이 이해

## 구조적 차이
```
[ 가상머신 (VM) ]                       [ 컨테이너 (Container) ]
+-------------------------------+       +-------------------------------+
|      애플리케이션 (App)         |       |      애플리케이션 (App)         |
+-------------------------------+       +-------------------------------+
|    라이브러리 / 바이너리        |       |    라이브러리 / 바이너리        |
+-------------------------------+       +-------------------------------+
|     게스트 OS (Guest OS)      |       |               -               |
+---------------^---------------+       +---------------^---------------+
|      하이퍼바이저 (Hypervisor)  |       |    컨테이너 엔진 (Docker/Podman)|
+---------------^---------------+       +---------------^---------------+
|      호스트 OS (Host OS)       |       |      호스트 OS (Host OS)       |
+---------------^---------------+       +---------------^---------------+
|      하드웨어 (H/W)            |       |      하드웨어 (H/W)            |
+-------------------------------+       +-------------------------------+
```

### 가상머신(VM)의 구조 - 수직적 고정
```
[ 유저 공간 ]   [ VM #1 (App) ]      [ VM #2 (App) ]      [ VM #3 (App) ]
                +-----------+        +-----------+        +-----------+
                | Guest OS  |        | Guest OS  |        | Guest OS  |
                +-----------+        +-----------+        +-----------+
----------------|-----------|--------|-----------|--------|-----------|---------
[ 가상 자원 ]   |   vCPU    |        |   vCPU    |        |   vCPU    |
(추상화 계층)   |   vRAM    |        |   vRAM    |        |   vRAM    |
                |   vDisk   |        |   vDisk   |        |   vDisk   |
----------------+-----------+--------+-----------+--------+-----------+---------
[ 하이퍼바이저 ]|               자원 에뮬레이션 및 스케줄링                     |
----------------+---------------------------------------------------------------+
[ 물리 자원 ]   |    [CPU]    [Memory]    [Disk/Storage]    [Network Card]    |
----------------+---------------------------------------------------------------+
```
- 특징 : 자원이 수직적으로 고정 -> 특정 VM이 메모리를 남겨도 다른 VM이 가져다 쓸 수 없음  
    - CPU/메모리 : 하드웨어 수준에서 설정한 메모리 주소 범위 전체 점유
    - 네트워크 : 각 VM이 물리 랜카드에 연결된 독립된 가상 컴퓨터처럼 동작 (자체적인 IP 스택 보유)
  
### 컨테이너 구조 - 울타리 정도의 고정
```
[ 유저 공간 ]   [Cont #1]  [Cont #2]  [Cont #3]  [Cont #4]  [Cont #5]  ...
                  (App)      (App)      (App)      (App)      (App)
                +-------+  +-------+  +-------+  +-------+  +-------+
                | Lib/B |  | Lib/B |  | Lib/B |  | Lib/B |  | Lib/B |
----------------+-------+--+-------+--+-------+--+-------+--+-------+---------
[ 커널 공간 ]   |  [ Namespace (격리) ]  |  [ cgroups (자원 제한) ]      |
(격리 계층)     |  - PID/Network/Mount   |  - CPU/Mem/IO Limits         |
----------------+-----------------------+-------------------------------+---------
[ 호스트 OS ]   |               단일 호스트 커널 (Shared Kernel)                |
----------------+---------------------------------------------------------------+
[ 물리 자원 ]   |    [CPU]    [Memory]    [Disk/Storage]    [Network Card]    |
----------------+---------------------------------------------------------------+
```
- 특징 : 자원이 수평적으로 유연하게 공유 -> 자원 경쟁 시 우선순위대로 분배 
  - CPU/메모리 : 커널이 직접 관리 + 컨테이너가 요청하는 즉시 물리 자원 할당
  - 디스크 : 레이어드 파일시스템을 통해 공통된 라이브러리는 하단에 적용하고 그 위에서 각자의 바뀐 부분 적용
  - 네트워크 : 호스트의 네트워크 스 안에서 veth라는 가상 파이프를 통해 데이터를 주고 받음 
    - `veth` (Virtual Ethernet Pair) : 쌍으로 생성되어 양쪽 끝이 연결된 파이프처럼 동작 


## 컨테이너의 격리 방법

### 네임스페이스 - 조회 가능 범위 제어
- PID : 컨테이너 내부에서 PID 1번으로 시작 (호스트의 다른 프로세스를 볼 수 없음)
- NET : 독립된 IP, 포트, 라우팅 테이블 보유
- MNT : 컨테이너 만의 독립된 파일 시스템 마운트 지점 보유 (호스트 경로 모름)
- UTS : 컨테이너마다 고유한 호스트 이름을 보유 가능
- IPC : 프로세스 간 통신 통로 분리 -> 다른 컨테이너와의 메시지 혼선 방지
- User : 컨테이너 안의 Root가 호스트에서는 일반 사용자로 매핑

### Control Group (cgroup) - 자원 사용 범위
- CPU : CPU 사용 시간이나 코어 수 제한
- Memory : 컨테이너가 사용할 수 있는 최대 메모리 설정 (초과 시 OOM Kill)
- I/O : 디스크 읽기쓰기 속도를 조절하여 특정 컨테이너의 자원 독점 방지
- PIDs : 컨테이너 안에서 생성할 수 있는 최대 프로세스 개수 제한
