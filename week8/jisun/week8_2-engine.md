# Docker와 Podman의 개념 및 기본 사용법

## 컨테이너 관리 도구 개념

### Docker : 중앙 집중형 관리
- 컨테이너 생명주기 전체를 하나의 거대 엔진이 책임지는 플랫폼

### Podman : 모듈화
- 컨테이너를 일반 리눅스 프로세스와 동일하게 취급하는 도구

### RHEL 환경에서의 Podman 채택 이유
- Docker가 root 권한으로 실행되는 중앙 데몬에 의존하여 해킹 시 위험  
  -> rootless 기능으로 기본 설계
- systemd가 컨테이너 프로세스 직접 제어 불가능  
    -> 컨테이너를 일반 리눅스 프로세스로 실행

- 쿠버네티스와의 연계


## 구조적 차이
```
[ Docker: Client-Server 모델 ]                 [ Podman: Fork-Exec 모델 ]
      
   사용자 (User)                                  사용자 (User)
       |                                              |
+------v-------+                               +------v-------+
|  Docker CLI  |                               |  Podman CLI  |
+------|-------+                               +------|-------+
       | (1) REST API 전송                            | (1) 자식 프로세스 생성
       |     (Unix Socket)                            |     (Fork / Exec)
       v                                              v
+--------------+                               +--------------+
| Docker Daemon| (2) 관리/모니터링                |  Conmon      | (2) 컨테이너 감시 전용
|  (dockerd)   | <----------------+            |  (프로세스)    | <--------------+
+------|-------+                  |            +------|-------+                |
       | (3) 컨테이너 생성           |                   | (3) 컨테이너 실행         |
       v                          |                   v                        |
+--------------+           [리소스 고정 점유]     +--------------+          [프로세스 생성 시만]
| Container A  |           (부하 발생 가능)       | Container A  |          (유연한 자원 소모)
+--------------+                               +--------------+
```
### Docker Daemon (dockerd)
- 실제 API 서버처럼 동작  
  - dockerd는 로컬 유닉스 소켓(`/var/run/docker.sock`)을 열고 대기  
    -> CLI에서 명령어 작성 시 HTTP 프로토콜로 컨테이너 생성 API 호출

- 컨테이너가 없어도 메모리에 상주 (기본 리소스 소모)  
- 모든 컨테이너의 로그, 네트워크, 이미지 관리 처리  
  -> 컨테이너 수 증가 시 데몬 프로세스 부하 증가 

### Podman
- API 서버를 거치지 않고 리눅스 커널 기능 직접 사용
  - Podman CLI가 직접 컨테이너 런타임 등을 실행하여 컨테이너 생성 등 진행

- Common (Container Monitor) : 데몬이 없는 대신 각 컨테이너마다 common이라는 작은 프로세스 부착  
-> 로그 저장과 컨테이너 상태 감시 담당 (이후 컨테이너가 죽으면 같이 사라짐)

- 각 컨테이너가 일반 프로세스로 취급되어 수평적으로 자원 분배 진행 

## Podman만의 특징

### Rootless 모드
```
[ 호스트 OS 레벨 ]                [ 컨테이너 내부 레벨 ]
(실제 커널이 보는 권한)              (프로세스가 믿는 권한)

  UID 1000 (User A)  <----------->  UID 0 (Root)
  UID 1001 (User A+1) <---------->  UID 1 (Bin)
  ...                               ...
  UID 65535 (User A+N) <--------->  UID 64535
```
- User Namespace 기술을 사용하여 호스트의 일반 사용자 ID를 컨테이너 내부의 Root ID로 매핑  
  -> 컨테이너 내부에서 root 권한이 탈취되더라도 호스트에서는 보안 유지 가능 

### Pod 개념
```
+---------------------------------------+
|              Pod (울타리)              |
|  +---------------------------------+  |
|  |     Infra Container (Pause)     |  | <--- 네트워크/포트/IP 점유
|  +---------------------------------+  |
|       |                     |         |
|  +----v----+           +----v----+    |
|  |  App    |           |  DB     |    | <--- 동일한 localhost 사용
|  |  Cont   | <-------> |  Cont   |    |      (서로 이름으로 통신)
|  +---------+           +---------+    |
+---------------------------------------+
```
- 한 Pod 안에 들어있는 컨테이너들은 동일한 네트워크 네임스페이스 공유  
    -> 같은 Pod 내의 컨테이너는 다른 컨테이너를 localhost로 접근 가능

## 기본 사용 법

### Dockeer 명령어
``` bash
$ docker run -d --name [이름] [이미지]
```
- 컨테이너 실행  
  - `-d` : 백그라운드에서 실행

``` bash
$ docker ps -a 
```
- 실행 중이거나 정지된 모든 컨테이너 목록 출력  
  - `-a` : 모든 컨테이너 출력

``` bash
$ docker images
```
- 로컬에 다운로드된 이미지 목록 확인

``` bash
$ docker stop [ID/이름]
```
- 실행 중인 컨테이너에 종료 신호를 보냄

``` bash
$ docker rm [ID/이름]
```
- 중지된 컨테이너를 시스템에서 완전히 삭제

``` bash
$ docker build -t [이름] .
```
- Dockerfile을 읽어 새로운 이미지 생성


### Podman 명령어
- 기본 명령어의 경우 docker와 동일하게 동작

``` bash
$ podman pod create
```
- 여러 컨테이너를 묶을 수 있는 Pod 공간 생성

``` bash
$ podman pod ps
```
- 현재 생성된 Pod 목록과 상태 확인

``` bash
$ podman generate systemd
```
- 컨테이너를 시스템 서비스 유닛 파일로 자동 변환

``` bash
$ podman login [레지스트리]
```
- 특정 저장소 등에 인증 정보 로그인
